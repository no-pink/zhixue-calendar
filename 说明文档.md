# 智学日程 — 技术说明文档

## 一、项目概述

智学日程是一款帮助用户制定每日学习计划的应用，以"计划→日历→小时任务"为主线，
支持多用户独立使用，数据持久化存储。

**技术栈**

| 层 | 技术 |
|---|---|
| 前端 | React 18 + Vite 6 + Tailwind CSS 4 |
| 后端 | Node.js + Express 4 |
| 数据库 | SQLite (better-sqlite3) |
| 认证 | JWT (jsonwebtoken) |
| 文件存储 | 本地磁盘 (server/uploads/) |

## 三、后端服务详解

### 3.1 启动方式

```bash
cd server
npm install        # 首次运行需要安装依赖
node index.js      # 启动服务
```

默认监听 **http://localhost:3001**，通过环境变量 `PORT` 可修改端口。

### 3.2 数据库 (SQLite)

**文件位置**: `server/data.db`（自动生成）

数据库包含 4 张表：

| 表名 | 用途 | 关键字段 |
|---|---|---|
| `users` | 用户账户 | id, username, password (bcrypt 加密), created_at |
| `plans` | 学习计划 | id, user_id (外键), name, start_date, end_date, created_at |
| `tasks` | 小时级任务 | id, plan_id (外键), date, hour (0-23), description, completed, created_at |
| `submissions` | 提交物 | id, task_id (外键), type (text/image/file/link), content, file_path, created_at |

**外键约束**:
- plans.user_id → users.id (CASCADE DELETE)
- tasks.plan_id → plans.id (CASCADE DELETE)
- submissions.task_id → tasks.id (CASCADE DELETE)

删除计划时，其下所有任务和提交物**自动级联删除**。

### 3.3 文件存储

**位置**: `server/uploads/`

- 用户上传的图片和文件以 UUID 重命名存储在此目录
- 文件路径记录在 `submissions.file_path` 字段
- 通过 `http://localhost:3001/uploads/{filename}` 访问
- **不会被 git 追踪**（.gitignore 中排除，仅保留 .gitkeep 占位）

### 3.4 API 接口列表

| 方法 | 路径 | 说明 | 认证 |
|---|---|---|---|
| POST | /api/auth/register | 注册 | 否 |
| POST | /api/auth/login | 登录 | 否 |
| POST | /api/auth/change-password | 修改密码 | 是 |
| GET | /api/plans | 获取当前用户的计划列表 | 是 |
| POST | /api/plans | 创建计划 | 是 |
| PUT | /api/plans/:id | 编辑计划 | 是 |
| DELETE | /api/plans/:id | 删除计划（含级联任务） | 是 |
| GET | /api/plans/:id/calendar | 获取日历统计数据 | 是 |
| GET | /api/tasks/:planId/:date | 获取某天所有任务 | 是 |
| POST | /api/tasks | 创建任务 | 是 |
| PUT | /api/tasks/:id | 更新任务/提交物 | 是 |
| PATCH | /api/tasks/:id/toggle | 切换完成状态 | 是 |
| DELETE | /api/tasks/:id | 删除任务 | 是 |
| POST | /api/tasks/upload | 上传文件 | 是 |
| POST | /api/tasks/batch | 批量填充任务 | 是 |
| GET | /api/backup/export | 导出全部数据（zip） | 是 |
| POST | /api/backup/restore | 导入恢复数据 | 是 |

JWT Token 通过 `Authorization: Bearer <token>` 请求头发送，有效期 **7 天**。

### 3.5 用户认证机制

- 密码使用 **bcrypt** 加盐哈希存储，数据库中不存明文
- JWT 密钥默认硬编码（`zhixue-calendar-secret-key-2024`），生产环境应通过环境变量 `JWT_SECRET` 设置
- Token 存储在客户端 `localStorage`，服务端无状态
- 每次 API 请求通过 `auth` 中间件验证 Token 并提取用户 ID

## 四、前端详解

### 4.1 启动方式

```bash
cd client
npm install        # 首次运行需要安装依赖
npx vite --host    # 启动开发服务器
```

默认监听 **http://localhost:5173**。

开发模式下，Vite 通过 proxy 将 `/api` 请求转发到后端 `localhost:3001`，因此前端无需配置跨域。

### 4.2 页面结构

```
/login              → 登录/注册页面
/                   → 主界面（受保护，需登录）
/*                  → 重定向到 /
```

### 4.3 组件逻辑

- **Login.jsx**: 登录与注册切换，成功后将 token 存入 localStorage 并跳转主界面
- **Dashboard.jsx**: 主框架。左侧 288px 计划列表，中间日历区域，右侧 384px 任务面板
- **PlanList.jsx**: 展示卡片式计划列表，支持展开创建/编辑表单
- **CalendarView.jsx**: 月历组件，支持月份切换、Ctrl多选日期、批量填充入口
- **TaskPanel.jsx**: 24小时时间轴任务面板，每个任务可勾选/编辑/删除/添加提交物
- **BatchFillModal.jsx**: 弹窗配置时间段和模板，处理冲突（跳过/覆盖）
- **SettingsModal.jsx**: 修改密码 + 数据导出/导入界面

### 4.4 用户状态

- Token 存储在 `localStorage['token']`
- 用户状态通过 React Context (`AuthContext`) 全局管理
- 页面刷新后自动检查 localStorage 恢复登录状态
- 关闭浏览器后再次打开，只要 token 未过期（7天）仍保持登录

## 五、数据存储与持久化说明

### 5.1 数据都存在哪里？

| 数据类型 | 存储位置 | 说明 |
|---|---|---|
| 用户账号信息 | `server/data.db` — users 表 | SQLite 数据库文件 |
| 计划数据 | `server/data.db` — plans 表 | 同上 |
| 任务数据 | `server/data.db` — tasks 表 | 同上 |
| 提交物文本/链接 | `server/data.db` — submissions 表 | 同上 |
| 上传的图片/文件 | `server/uploads/` | 物理文件，路径记录在数据库 |
| 用户登录 Token | 浏览器 `localStorage` | 仅在浏览器端，服务端无状态 |
| 用户会话状态 | React Context（内存） | 页面刷新从 localStorage 恢复 |

### 5.2 数据持久性

| 操作 | 影响 |
|---|---|
| 刷新页面 (F5) | 前端状态重置，Context 重新加载，但 token 仍在 localStorage，API 可正常调用 |
| 关闭浏览器标签页 | 同上，重新打开网站仍可保持登录 |
| 清除浏览器缓存/Cookie | Token 从 localStorage 丢失，需要重新登录。**数据库数据不丢失** |
| 更换浏览器/设备 | 需要重新登录。数据库数据完好 |
| **关闭后端服务** | 前端页面可打开，但所有 API 调用失败（白屏/报错）。**数据不丢失** |
| **重启后端服务** | 一切恢复正常，所有数据完好 |
| 删除 `server/data.db` | **所有数据永久丢失**（除非有备份） |
| 删除 `server/uploads/` | 上传文件丢失，但数据库中的文字数据仍在 |

### 5.3 关键结论

> **数据保存在服务端 `data.db` 文件和 `uploads/` 目录中，与浏览器无关。**
>
> 删除浏览器缓存、Cookie、更换浏览器、甚至更换电脑，
> 只要 `server/data.db` 文件还在，登录后数据就还在。

## 六、启动与关闭的影响

### 6.1 启动顺序

1. **先启动后端** (`node server/index.js`)
   - 自动创建 `data.db` 和 `uploads/` 目录
   - 自动建表（如果表不存在）
2. **再启动前端** (`npx vite`)
   - 代理/api到后端，页面可正常使用

### 6.2 关闭后端的影响

- 前端页面仍在，但所有功能不可用（无法登录、看不到数据）
- 数据**不丢失**
- 重新启动后端后一切恢复正常

### 6.3 同时关闭两者的影响

- 前端无法访问，后端数据完好
- 下次启动后正常使用

### 6.4 迁移到其他机器

只需复制整个 `server/` 目录（或至少 `data.db` + `uploads/`）到新机器，
在新机器上 `npm install && node index.js` 即可无缝切换。

## 七、多用户数据隔离

- 每个用户有独立的 `user_id`
- `plans` 表通过 `user_id` 外键关联用户
- 所有 API 路由中的 `auth` 中间件提取当前登录用户 ID
- 查询语句**全部**带有 `AND user_id = ?` 条件
- 用户 A 登录后**完全看不到**用户 B 的任何数据
- 数据库层面、API 层面双重隔离

## 八、备份与恢复

### 8.1 导出

- 进入"设置 → 数据备份 → 导出数据"
- 下载一个 `.zip` 包，包含：
  - `data.json`：全部计划、任务、提交物的 JSON 数据
  - `files/` 目录：所有上传的文件实体
- 此 zip 包可脱离本系统独立保管

### 8.2 导入

- 进入"设置 → 数据备份 → 导入数据"
- 选择之前导出的 `.zip` 或 `.json` 文件
- 系统会**完全替换**当前用户的所有数据（含文件）
- 操作前有确认弹窗，需用户手动确认

### 8.3 备份建议

- 定期导出一份 zip 包保存到其他位置
- 直接备份 `server/data.db` + `server/uploads/` 是更完整的方案

## 九、生产环境部署建议

当前为开发模式，如需部署到生产环境：

| 注意事项 | 说明 |
|---|---|
| JWT 密钥 | 通过环境变量 `JWT_SECRET` 设置强密码，不要使用硬编码默认值 |
| 端口 | 通过 `PORT` 环境变量调整 |
| 前端构建 | `cd client && npx vite build` 生成 `dist/`，用 Nginx 或 CDN 托管 |
| CORS | 生产环境建议将 `cors()` 配置为具体域名而非允许所有来源 |
| 数据库并发 | SQLite WAL 模式已开启，高并发场景可考虑迁移至 PostgreSQL |
| 文件存储 | `uploads/` 可挂载云存储（OSS/S3）或使用反向代理 |
| HTTPS | 生产环境必须配置 HTTPS |
| 进程管理 | 使用 `pm2`、`systemd` 或 Docker 管理后端进程 |
| 日志 | 建议接入日志系统（winston/pino），`console.error` 在 pm2 下会丢失 |

## 十、常见问题

**Q: 忘记密码怎么办？**
目前不支持密码重置。可在 `server/data.db` 中手动删除用户行后重新注册，
或通过数据库的 SQLite CLI 修改。

**Q: 上传的文件存在哪里？存在多久？**
存在 `server/uploads/` 中，永久保留直到手动删除或随计划级联删除。

**Q: 最多支持多少用户？多少数据？**
SQLite 没有硬性用户数上限，单文件可支持 TB 级数据。
但也受磁盘空间限制，尤其是上传的图片/文件。

**Q: 数据库文件损坏了怎么办？**
SQLite 使用 WAL 模式有一定崩溃恢复能力。建议定期导出 zip 备份。
如果 `data.db` 损坏，可用 zip 中的 `data.json` 重新导入恢复。

**Q: 可以多人在线同时使用吗？**
可以，SQLite 的 WAL 模式支持并发读，写操作会串行化。
但 SQLite 不适合高并发写入场景（数十人同时操作）。此场景建议换用 PostgreSQL。
