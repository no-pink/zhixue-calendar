# 未来优化计划

> 本文档记录已识别但尚未实施的优化项，按优先级排列。
> 生成日期：2026-07-14

---

## P0 — 高价值、低成本

### 1. 复制任务集成日历日期多选

**问题**：CopyTasksModal 目前只能手动输入日期，每次只能添加一天，跨多天复制操作繁琐。

**方案**：在弹窗中嵌入一个精简日历组件，支持点选多个日期（复用 CalendarView 的多选逻辑）。

**工时**：2-3 小时

---

### 2. 后端自动化测试

**问题**：0 测试覆盖，每次改动需要手动验证。

**方案**：
- 使用 `vitest` + `supertest`，重点覆盖：
  - auth 路由（登录/注册/改密/限速）
  - tasks CRUD（含冲突检测逻辑）
  - batch/copy 端点
  - backup 导出/恢复

**工时**：4-6 小时

---

### 3. ESLint 代码规范

**问题**：无代码规范检查，代码风格不一致。

**方案**：在 `client/` 和 `server/` 分别初始化 ESLint，使用 `eslint-config-prettier`。加入 pre-commit hook 或 CI 流程。

**工时**：1 小时（配置）+ 0.5 小时（修复初始违规）

---

## P1 — 中价值、中成本

### 4. 错误码标准化

**问题**：后端错误返回格式不统一——有的是 `{ error: "xxx" }`，有的是 `{ error: "xxx", conflict: true }`，前端需要各种分支判断。

**方案**：统一为 `{ code: string, message: string, details?: any }`：
- `AUTH_EXPIRED` / `AUTH_REQUIRED` / `NOT_FOUND` / `CONFLICT_SAME_NAME` / `CONFLICT_OVERLAP` / `VALIDATION_ERROR`
- 前端用 `code` 做逻辑判断，`message` 显示给用户

**工时**：3-4 小时

---

### 5. 提交物图标 SVG 化

**问题**：当前使用 emoji（📝🔗📎），在不同操作系统和浏览器上渲染差异大。

**方案**：将 emoji 替换为内联 SVG 图标，保持视觉一致性。

**工时**：1 小时

---

### 6. 快捷键一致性

**问题**：
- Enter 在登录页和工作正常，但在编辑表单有时不能提交
- Escape 在部分弹窗中关闭，在部分弹窗中无效
- 没有全局键盘导航（如 ← → 切换日期）

**方案**：
- 所有表单绑定 Enter 提交、Escape 取消/关闭
- CalendarView 添加 ← → 键左右切换月份
- 任务面板绑定 `j/k` 上下导航

**工时**：2-3 小时

---

## P2 — 高价值、高成本

### 7. 任务拖拽调整时段

**问题**：调整任务时间需要点击编辑→修改下拉框→确定，三步操作。拖拽更直观。

**方案**：在 TaskItem 的时间标签上添加拖拽句柄，拖动可调整 start_hour/end_hour。
使用 `dnd-kit` 或原生 HTML5 Drag & Drop API。

**工时**：4-6 小时

---

### 8. 计划仪表盘

**问题**：目前进度展示方式单一，缺乏整体趋势感知。

**方案**：在 Dashboard 中添加仪表盘区域（或在 CalendarView 上方）：
- 本周完成率折线图
- 各计划进度对比
- 连续学习天数（Streak）

使用 `recharts` 或内联 SVG。

**工时**：6-8 小时

---

### 9. 移动端手势适配

**问题**：日历视图在手机上只能靠点击按钮切换月份，不支持滑动。

**方案**：
- 日历区域支持左/右滑切换月份（`touchstart`/`touchend` 检测横滑距离）
- 任务列表支持下拉刷新
- 侧边栏支持右滑打开

**工时**：3-4 小时

---

### 10. 数据看板（管理员视角）

**问题**：用户无法看到自己的学习统计，如总任务数、完成率趋势、平均每日任务量。

**方案**：新增统计 API `GET /api/plans/stats`，前端展示卡片式看板：
- 总任务 / 已完成 / 完成率
- 本月趋势（每日完成数）
- 最活跃时段分布

**工时**：4-5 小时

---

## 技术债务

### 11. 前端 api/index.js 按 domain 拆分

**问题**：当前 70 行，但随着社交功能加入会快速膨胀。

**方案**：拆分为 `api/auth.js` `api/plans.js` `api/tasks.js` `api/backup.js`。

**时机**：新功能开发前拆分，不单独占工时。

---

### 12. plans.js 路由提取 service 层

**问题**：plans.js 的路由函数中内含 SQL 逻辑，与优化前的 tasks.js 类似。

**方案**：参照 `services/taskService.js` 模式，提取 `services/planService.js`。

**工时**：1 小时

---

### 13. 前端组件 Storybook / 组件文档

**问题**：随着组件增多，缺乏独立预览和文档。

**方案**：初始化 Storybook，为每个弹窗组件和 TaskItem 等通用组件编写 story。

**工时**：3-4 小时（初始化）+ 0.5 小时/组件
