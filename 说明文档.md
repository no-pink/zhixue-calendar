# 智学日程 — 技术说明文档

> 最后更新：2026-07-15

## 一、项目概述

智学日程是一款帮助用户制定每日学习计划的应用，以"计划→日历→小时任务"为主线，
支持多用户独立使用，数据持久化存储。

**技术栈**

| 层 | 技术 |
|---|---|
| 前端 | React 18 + React Router 6 + Vite 6 + Tailwind CSS 4 |
| 后端 | Node.js + Express 4 |
| 数据库 | SQLite (better-sqlite3)，WAL 模式 |
| 认证 | JWT (jsonwebtoken) + bcryptjs 密码哈希 |
| 日志 | Pino（结构化日志） |
| 文件存储 | 本地磁盘 (server/uploads/)，UUID 重命名 |
| 安全头 | helmet（CSP / X-Frame-Options / HSTS 等） |
| 限速 | express-rate-limit（全局 100 次/15min，登录 10 次/15min） |
| 测试 | Mocha + Supertest + assert 严格模式 |

## 三、后端服务详解

### 3.1 启动方式

```bash
cd server
npm install        # 首次运行需要安装依赖
node index.js      # 启动服务，默认监听 http://localhost:3001
npm test           # 运行集成测试
```

通过环境变量 `PORT` 可修改端口。

### 3.2 数据库 (SQLite)

**文件位置**: `server/data.db`（自动生成，WAL 模式）

数据库包含 5 张表：

| 表名 | 用途 | 关键字段 |
|---|---|---|
| `users` | 用户账户 | id, username, password (bcrypt), created_at |
| `plans` | 学习计划 | id, user_id (FK), name, start_date, end_date, created_at |
| `tasks` | 小时级任务 | id, plan_id (FK), date, start_hour, end_hour, description, completed, created_at |
| `submissions` | 提交物 | id, task_id (FK), type (text/image/file/link), content, file_path, created_at |
| `schema_migrations` | 迁移记录 | version, name, applied_at |

**字段约束**:
- `start_hour`: INTEGER, CHECK(start_hour >= 0 AND start_hour < 24)
- `end_hour`: INTEGER, CHECK(end_hour > 0 AND end_hour <= 24)
- tasks 表兼存 `hour` 旧字段的兼容迁移（自动检测并升级为 start_hour/end_hour）

**外键级联删除**:
- plans.user_id → users.id (CASCADE DELETE)
- tasks.plan_id → plans.id (CASCADE DELETE)
- submissions.task_id → tasks.id (CASCADE DELETE)

删除计划时，其下所有任务和提交物**自动级联删除**。

**索引**:
- `idx_plans_user` ON plans(user_id)
- `idx_tasks_plan_date` ON tasks(plan_id, date)
- `idx_submissions_task` ON submissions(task_id)

### 3.3 数据库迁移

通过 `server/migrate.js` 管理。在 `server/migrations/` 目录下放置 `NNNN_description.sql` 文件（NNNN 为 4 位版本号），服务启动时自动检测并执行未应用的迁移，结果记录在 `schema_migrations` 表中。迁移在独立事务中执行，失败不影响已有数据。

### 3.4 文件存储

**位置**: `server/uploads/`

- 用户上传的图片和文件以 UUID 重命名存储在此目录
- 文件路径记录在 `submissions.file_path` 字段
- 通过 `http://localhost:3001/api/uploads/{filename}` 访问并映射为静态资源
- **不会被 git 追踪**（.gitignore 中排除，仅保留 .gitkeep 占位）
- 上传限制：文件大小 10MB（可配置），MIME 类型白名单校验

### 3.5 API 接口列表

所有业务接口需 `Authorization: Bearer <token>` 请求头。

| 方法 | 路径 | 说明 | 认证 | 限速 |
|------|------|------|------|------|
| POST | /api/auth/register | 注册 | 否 | 10次/15min |
| POST | /api/auth/login | 登录 | 否 | 10次/15min |
| POST | /api/auth/change-password | 修改密码 | 是 | — |
| GET | /api/auth/me | 获取当前用户 | 是 | — |
| GET | /api/plans | 获取计划列表（含进度统计） | 是 | — |
| POST | /api/plans | 创建计划（同名检测） | 是 | — |
| PUT | /api/plans/:id | 编辑计划 | 是 | — |
| DELETE | /api/plans/:id | 删除计划（级联删除） | 是 | — |
| GET | /api/plans/:id/calendar | 获取日历统计数据 | 是 | — |
| GET | /api/tasks/:planId/:date | 获取某天所有任务（含提交物） | 是 | — |
| POST | /api/tasks | 创建任务（冲突检测→三种模式） | 是 | — |
| PUT | /api/tasks/:id | 更新任务/提交物（检测冲突） | 是 | — |
| PATCH | /api/tasks/:id/toggle | 切换完成状态 | 是 | — |
| DELETE | /api/tasks/:id | 删除任务（清理上传文件） | 是 | — |
| POST | /api/tasks/upload | 上传文件（MIME + 大小校验） | 是 | — |
| POST | /api/tasks/batch | 批量填充（多时段 × 多日期） | 是 | — |
| POST | /api/tasks/batch-simple | 简单批量创建（同一时段 × 多日期） | 是 | — |
| POST | /api/tasks/copy | 复制任务到其他日期/计划 | 是 | — |
| GET | /api/backup/export | 导出数据 ZIP（含 JSON + 附件） | 是 | — |
| POST | /api/backup/restore | 恢复数据（完全替换当前用户数据） | 是 | — |

### 3.6 错误码体系

统一格式 `{ code: string, message: string, details?: any }`：

| Code | HTTP 状态 | 含义 |
|------|----------|------|
| `AUTH_REQUIRED` | 401 | 未提供 Token |
| `AUTH_EXPIRED` | 401 | Token 无效或已过期 |
| `AUTH_FAILED` | 400 | 用户名或密码错误 |
| `VALIDATION_ERROR` | 400 | 请求参数不合法 |
| `CONFLICT` | 400 | 资源冲突（如重名计划） |
| `NOT_FOUND` | 404 | 资源不存在 |
| `RATE_LIMITED` | 429 | 请求频率超限 |
| `INTERNAL_ERROR` | 500 | 服务器内部异常 |
| `CONFLICT_SAME_NAME` | 200 | 任务同名冲突（前端弹窗处理） |
| `CONFLICT_OVERLAP` | 200 | 任务时段重叠（前端弹窗处理） |

### 3.7 用户认证机制

- 密码使用 **bcrypt** 加盐哈希（10 轮）存储，数据库中不存明文
- JWT 密钥：优先读取环境变量 `JWT_SECRET`，开发环境 fallback 为 `zhixue-calendar-dev-secret`，生产环境缺失时报错退出
- Token 有效期默认 7 天（可通过 `JWT_EXPIRES_IN` 配置）
- Token 存储在客户端 `localStorage`，服务端无状态
- 每次 API 请求通过 `auth` 中间件验证 Token 并提取 `{ id, username }`
- auth 中间件通过 `module.exports.auth = auth` 导出供其他模块复用

### 3.8 后端架构分层

```
routes/         →  编排层：解析请求、调用 service、返回响应
services/       →  业务逻辑层：冲突检测、CRUD、聚合查询
db.js           →  数据访问层：数据库连接、建表、索引
errors.js       →  错误处理层：统一错误码和响应格式
config.js       →  配置层：环境变量集中管理
logger.js       →  日志层：Pino 结构化日志
migrate.js      →  迁移层：版本化数据库变更
```

### 4.3 组件关系

```
App.jsx
├── AuthProvider (Context)
│   └── ToastProvider (Context)
│       └── Login.jsx                    — /login 路由
│       └── Dashboard.jsx                — / 路由
│           ├── PlanList.jsx             — 左侧：计划 CRUD
│           ├── CalendarView.jsx         — 中间：月历 + 多选
```

### 4.4 用户状态管理

- Token 存储在 `localStorage['token']`
- 用户状态通过 React Context (`AuthContext`) 全局管理
- 页面刷新后从 localStorage 恢复（JWT 解码获取 username，无需额外 API 调用）
- 关闭浏览器后再次打开，只要 token 未过期（7天）仍保持登录
- `AuthContext` 提供：`user`, `loading`, `login()`, `register()`, `logout()`, `changePassword()`

### 4.5 Toast 通知

- `ToastContext` 提供 `toast.success(msg)`, `toast.error(msg)`, `toast.info(msg)`
- 自动 3 秒后消失，可自定义 duration
- 固定定位在页面右上角，带 slide-in 动画
- 支持手动点击关闭

### 4.6 响应式设计

| 断点 | 布局 |
|------|------|
| >= 768px (md) | 三栏：计划列表 (240-288px) + 日历 (flex-1) + 任务面板 (320-384px) |
| < 768px | 日历全宽纵向列表 + 侧边栏抽屉 + 任务面板浮动弹窗 |

移动端日历已改为纵向按日列表展示（替代之前固定 490px 网格），桌面端保持 7 列网格。

### 4.7 冲突处理流程

```
创建/编辑任务 → API 返回 { code: "CONFLICT_SAME_NAME" | "CONFLICT_OVERLAP" }
             → TaskConflictDialog 弹窗
             → 用户选择: 保留新旧 / 跳过 / 覆盖
             → 重新调用 API 并带上 force 或 conflict_mode 参数
             → 完成操作
```

## 五、数据存储与持久化说明

### 5.1 数据存储位置

| 数据类型 | 存储位置 | 说明 |
|---|---|---|
| 用户账号信息 | `server/data.db` — users 表 | SQLite，密码 bcrypt 哈希 |
| 计划数据 | `server/data.db` — plans 表 | 关联 user_id |
| 任务数据 | `server/data.db` — tasks 表 | 关联 plan_id，start_hour/end_hour 范围 0-24 |
| 提交物文本/链接 | `server/data.db` — submissions 表 | 关联 task_id |
| 上传的图片/文件 | `server/uploads/` | UUID 重命名，路径记录在 submissions.file_path |
| 用户登录 Token | 浏览器 `localStorage` | 仅在浏览器端，服务端无状态 |
| 用户会话状态 | React Context（内存） | 页面刷新从 localStorage 恢复 |

### 5.2 数据持久性

| 操作 | 影响 |
|---|---|
| 刷新页面 (F5) | 前端状态重置，Context 重新加载，token 仍在 localStorage，API 正常调用 |
| 关闭浏览器标签页 | 同上，重新打开网站仍可保持登录 |
| 清除浏览器缓存/localStorage | Token 丢失，需重新登录。**数据库数据不丢失** |
| 更换浏览器/设备 | 需要重新登录。数据库数据完好 |
| **关闭后端服务** | 前端页面可打开，所有 API 调用失败。**数据不丢失** |
| **重启后端服务** | 一切恢复正常，所有数据完好 |
| 删除 `server/data.db` | **所有数据永久丢失**（除非有备份） |
| 删除 `server/uploads/` | 上传文件丢失，但数据库中的文字数据仍在 |

> **数据保存在服务端 `data.db` 文件和 `uploads/` 目录中，与浏览器无关。**

## 七、多用户数据隔离

- 每个用户有独立的 `user_id`
- `plans` 表通过 `user_id` 外键关联用户
- 所有 API 路由中的 `auth` 中间件提取当前登录用户 ID
- 查询语句**全部**带有 `AND user_id = ?` 条件
- `verifyPlanAccess()` 和 `verifyTaskOwnership()` 提供二次校验
- 用户 A 登录后**完全看不到**用户 B 的任何数据
- 数据库层面、API 层面双重隔离

## 八、备份与恢复

### 8.1 导出

- 进入"设置 → 数据备份 → 导出数据"
- 下载一个 `.zip` 包，包含：
  - `data.json`：全部计划、任务、提交物的 JSON 数据
  - `files/` 目录：所有上传的文件实体
- 此 zip 包可脱离本系统独立保管

### 8.2 导入

- 进入"设置 → 数据备份 → 导入数据"
- 选择**之前导出的 `.zip` 文件**（需先解压取出 `data.json`）或直接选 `.json` 文件
- 系统会**完全替换**当前用户的所有数据（含文件）
- 操作前有确认弹窗，需用户手动确认
- 导入后自动刷新页面

> **注意**：当前导入 ZIP 需要用户先解压。未来将支持直接导入 ZIP。

### 8.3 备份建议

- 定期导出一份 zip 包保存到其他位置
- 直接备份 `server/data.db` + `server/uploads/` 是更完整的方案

## 九、生产环境部署建议

### 前置准备

部署前请确认以下 4 项已完成：

1. **JWT 密钥**：通过环境变量 `JWT_SECRET` 设置强随机密钥（`NODE_ENV=production` 时缺失会拒绝启动）
2. **CORS 限制**：`CORS_ORIGIN` 设为你的域名（生产环境 `*` 会打印警告）
3. **前端 API 地址**：构建前端时设 `VITE_API_BASE_URL` 环境变量（同域反代可留空，跨域需设完整地址）
4. **helmet**：已集成 helmet 安全头中间件

### 推荐部署架构

```
用户 → Nginx (:80/443) → 前端静态文件 (dist/)
                        → /api/* → 127.0.0.1:3001 (Express)
```

| 注意事项 | 说明 |
|---|---|
| JWT 密钥 | 必须通过环境变量 `JWT_SECRET` 设置强密码，缺失时生产环境拒绝启动 |
| 端口 | 通过 `PORT` 环境变量调整 |
| 前端构建 | `cd client && npx vite build` 生成 `dist/`，用 Nginx 或 CDN 托管 |
| CORS | 生产环境建议将 `CORS_ORIGIN` 配置为具体域名 |
| 数据库并发 | SQLite WAL 模式已开启，高并发场景可考虑迁移至 PostgreSQL |
| 文件存储 | `uploads/` 可挂载云存储（OSS/S3）或使用反向代理 |
| HTTPS | 生产环境必须配置 HTTPS（建议 Nginx 反向代理终止 TLS） |
| 进程管理 | 使用 `pm2`、`systemd` 或 Docker 管理后端进程 |
| 安全头 | 已集成 helmet 中间件（CSP、HSTS 等） |
| 密码策略 | 建议增加密码最小长度校验（当前无限制） |

## 十、常见问题

**Q: 忘记密码怎么办？**
目前不支持密码重置。可在 `server/data.db` 中手动删除用户行后重新注册，
或通过 SQLite CLI 修改。

**Q: 上传的文件存在哪里？存在多久？**
存在 `server/uploads/` 中，永久保留直到手动删除或随计划级联删除。

**Q: 最多支持多少用户？多少数据？**
SQLite 没有硬性用户数上限，单文件可支持 TB 级数据。
但也受磁盘空间限制，尤其是上传的图片/文件。

**Q: 数据库文件损坏了怎么办？**
SQLite 使用 WAL 模式有一定崩溃恢复能力。建议定期导出 zip 备份。
如果 `data.db` 损坏，可用 zip 中的 `data.json` 重新导入恢复。

**Q: 可以多人在线同时使用吗？**
可以，SQLite 的 WAL 模式支持并发读，写操作会串行化。
但 SQLite 不适合高并发写入场景（数十人同时操作）。此场景建议换用 PostgreSQL。
