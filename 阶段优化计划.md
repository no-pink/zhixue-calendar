# 阶段优化计划

> 基于代码审查与项目评价整理的优化方向，按阶段和紧迫性排列。
> 最后更新：2026-07-14

---

## 第一阶段（基础设施补课）

这些是当前项目最薄弱的环节，不做就谈不上后续扩展。

### 1. 后端自动化测试

**问题**：0 测试覆盖。核心的冲突检测逻辑（同名/重叠/force/跨计划复制/批量填充）全靠手点验证，任何一次重构都可能在不知情的情况下破坏已有逻辑。

**方案**：
- 使用 `vitest` + `supertest`，模拟 SQLite 内存数据库
- 重点覆盖：
  - auth 路由（登录/注册/改密/限速表现）
  - tasks CRUD —— 尤其是 `findOverlaps` 和 `resolveTaskConflict` 的 4 个分支
  - batch / batch-simple / copy 端点
  - backup 导出/恢复（JSON 格式兼容性）
- 集成到 `npm test` 脚本，CI 自动运行

**工时**：6-8 小时

---

### 2. 错误码标准化

**问题**：后端错误返回格式混搭——正常错误 `{ error: "xxx" }`，冲突返回 `{ conflict: true, conflictType: "same_name", overlapping: [...] }`，前端需要 `if (res.conflict)` 分支判断。这种"按端点自由发挥"的做法在接口增多时会快速失控，前端每接一个新端点都要猜返回格式。

**方案**：统一为 `{ code: string, message: string, details?: any }`：
- `AUTH_EXPIRED` / `AUTH_REQUIRED` / `NOT_FOUND`
- `CONFLICT_SAME_NAME` / `CONFLICT_OVERLAP`
- `VALIDATION_ERROR` / `FORBIDDEN`
- 前端根据 `code` 做逻辑判断，`message` 做显示

**工时**：3-4 小时

---

### 3. 移动端日历突破 490px 限制

**问题**：CalendarView 依赖 `min-w-[490px]` 保证 7 列网格显示，在窄屏手机上用户被迫横向滚动——这不是"响应式"，只是加了一层遮羞布。真正的移动端日历应该在小屏上切换为紧凑布局或垂直列表。

**方案**：
- `<768px` 时切换到 5 列或按日滚动视图
- 或放弃网格，改为按日列表展示（类似手机系统日历的月视图）
- 保留左右滑动手势切换月份

**工时**：4-6 小时

---

## 第二阶段（体验打磨）

### 4. 移动端手势适配

**问题**：日历切换靠点击按钮，任务列表无下拉刷新，侧边栏必须精确点击汉堡图标才能打开。手机上操作手感生硬，和原生应用差距明显。

**方案**：
- 日历区域 `touchstart`/`touchend` 检测水平滑动，切换月份
- 任务列表下拉触发重新加载
- 侧边栏支持从屏幕左边缘右滑打开

**工时**：3-4 小时

---

### 5. 复制任务日期选择改进

**问题**：CopyTasksModal 靠手动一个个输入日期，选 10 天要添加 10 次，效率极低。

**方案**：嵌入迷你日历组件，支持点选多日（复用 CalendarView 的多选逻辑），同时保留手动输入作为补充。

**工时**：2-3 小时

---

### 6. 快捷键一致性

**问题**：
- Enter 提交 / Escape 关闭在部分表单中生效，部分不生效
- 无全局键盘导航（← → 切换月份、j/k 导航任务列表）
- 不同弹窗的实现方式不一致

**方案**：统一所有表单和弹窗的键盘事件处理，CalendarView 绑定方向键。

**工时**：1-2 小时

---

### 7. 提交物图标 SVG 化

**问题**：当前使用 emoji（📝🔗📎）渲染提交物类型标识。emoji 在不同操作系统上外观差异显著，且在高 DPI 屏幕下边缘模糊。

**方案**：替换为内联 SVG 图标（使用 Heroicons 或自绘），保持视觉一致性。

**工时**：1 小时

---

## 第三阶段（新功能模块）

### 8. 任务拖拽调整时段

**问题**：调整任务时间需要三步操作（点击编辑→改下拉框→确认），高频操作路径太长。

**方案**：在 TaskItem 的时间区域添加拖拽手柄，支持纵向拖动修改 start_hour/end_hour。使用 `dnd-kit` 库。

**工时**：4-6 小时

---

### 9. 学习仪表盘 & 数据统计

**问题**：用户对自己长期的学习趋势没有感知——看不到完成率变化、最活跃时段、连续学习天数。

**方案**：
- 后端 `GET /api/plans/stats` 聚合统计
- 前端 Dashboard 顶部区域展示卡片：
  - 总体完成率（环形进度）
  - 近 7 天完成趋势（迷你折线图）
  - 学习连续天数
  - 最活跃时间段分布（横向条形图）
- 使用 `recharts` 或纯 SVG

**工时**：6-8 小时

---

## 第四阶段（工程化 & 部署）

### 10. ESLint + Prettier

**问题**：代码风格靠肉眼审查，客户端和服务端的风格不完全统一（引号、分号、箭头函数格式等）。

**方案**：根目录统一配置 ESLint + Prettier，加入 pre-commit hook（husky + lint-staged）。

**工时**：1-2 小时

---

### 11. Docker 化 & CI/CD

**问题**：目前部署方式仍为手动 `node index.js`，没有容器化和自动部署流程，换环境或新成员加入需要手动配置。

**方案**：
- 根目录添加 `Dockerfile`（多阶段构建：前端 build → Nginx 静态服务 + Node 后端）
- `docker-compose.yml` 编排前后端
- GitHub Actions 自动构建镜像并推送

**工时**：3-4 小时

---

### 12. SQLite → PostgreSQL 迁移准备

**问题**：SQLite 的并发写性能在超过数个并发用户后会明显下降。如果后续推进社交功能或 AI 代理，必定成为瓶颈。但现阶段过早迁移也是过度工程。

**方案**：先不迁移，但做以下准备：
- 所有 SQL 查询保持标准 SQL，不使用 SQLite 专有语法
- `db.js` 抽象为接口，后续替换实现时不需改动 `services/` 层
- `config.js` 中 `dbPath` 改为 `dbConnection` 配置项

**时机**：当用户数 > 5 人或需要部署到团队使用时启动

**工时**：2-3 小时（仅准备阶段）

---

## 技术债务（随新功能消化）

### 13. plans.js 提取 service 层

**问题**：`routes/plans.js` 的路由函数中直接写 SQL 和业务逻辑，与优化前的 `tasks.js` 是同样的问题。尚未提取 service 层。

**方案**：参照 `services/taskService.js` 模式，提取 `services/planService.js`。

**工时**：1 小时

---

### 14. 前端 api/index.js 按 domain 拆分

**问题**：当前 ~80 行，尚可接受。但社交功能（好友/分享/动态）加入后会膨胀。

**方案**：拆分为 `api/auth.js` `api/plans.js` `api/tasks.js` `api/backup.js`。

**时机**：新功能开发前拆分，不单独占工时。

## 优先级总览

| 阶段 | 项数 | 预估工时 | 核心目标 |
|------|------|---------|---------|
| 第一阶段 | 3 | 13-18h | 补测试、统一错误处理、真·移动端适配 |
| 第二阶段 | 4 | 7-10h | 手势、选择效率、快捷键、视觉一致性 |
| 第三阶段 | 2 | 10-14h | 拖拽交互、数据洞察 |
| 第四阶段 | 3 | 6-9h | ESLint、Docker、数据库抽象 |
| 技术债务 | 2 | 1h | 随新功能消化 |
| **合计** | **14** | **37-52h** | |

### 建议执行策略

1. **第一阶段**是最值得投入的——没有测试和统一的错误处理，后续每加一个功能都可能引入回归。建议全力完成后再开始新功能开发。
2. **第二阶段**的移动端手势（#4）和日历突破 490px 限制（#3）有依赖关系，建议同批处理。
3. **第三阶段**的拖拽（#8）和仪表盘（#9）价值高但成本也高，建议在基础补课完成后选择性开始。
4. **第四阶段**的 Docker 和 CI/CD（#11）在需要部署上线前做即可，不宜过早投入。
