# 未来计划 — 智学日程演进路线

> 本文档评估当前架构的可扩展性，分析未来三大功能的可行性，并给出分阶段实施建议。
> 日期：2026-07-12

---

## 一、当前架构评估

### 现状（截至 2026-07-15）

| 层面 | 当前技术 | 定位 |
|------|---------|------|
| 前端 | React 18 SPA + Vite + Tailwind CSS 4 | 适合快速迭代，有 Toast/多选/冲突弹窗/拖拽复制等交互 |
| 后端 | Express 4 分层架构 | 路由编排层 + 业务逻辑层(services/) + 数据访问层(db.js) 已分层 |
| 数据库 | SQLite (better-sqlite3) + 迁移系统 | 零运维，WAL 模式，版本化 SQL 迁移自动执行 |
| 认证 | JWT + bcryptjs | 基础可用，缺乏 refresh token 和细粒度权限 |
| 错误处理 | 统一错误码 { code, message, details } | AppError 类 + sendError + sendConflict，前端按 code 判断 |
| 日志 | Pino 结构化日志 | 开发环境 pino-pretty 彩色输出，生产环境 JSON |
| 测试 | Mocha + Supertest（20 个集成测试） | 覆盖 auth/plans/tasks，backup 和改密尚未覆盖 |
| 实时通信 | 无 | 所有交互基于 HTTP 请求-响应 |
| 文件存储 | 本地文件系统 | 简单但受磁盘容量限制 |
| 代码质量 | ESLint/Prettier 尚未引入 | 代码风格靠人工审查 |

### 已完成的架构改进（相比初版）

1. **后端分层** — `services/taskService.js` 从路由中抽离业务逻辑，路由只做编排
2. **错误码标准化** — 统一 `{ code, message, details }` 格式，`server/errors.js` 集中管理
3. **数据库迁移系统** — `server/migrate.js` + `migrations/` 目录，版本化增量变更
4. **结构化日志** — 从 `console.log` 迁移到 Pino
5. **集中配置** — `server/config.js` 统一管理所有环境变量
6. **集成测试** — 20 个测试用例覆盖核心路由

### 可扩展性结论

**短期（1-3 个月）**：当前架构可以支撑新功能的 MVP 版本，但需要关键改造。

**长期（6 个月以上）**：以下瓶颈必须解决：
- SQLite → PostgreSQL（并发、全文搜索、JSON 查询）
- 单体 → 模块化拆分（至少将 AI 和推荐抽离为独立服务）
- 无队列 → 引入任务队列（AI 请求耗时较长）
- 无缓存 → 引入 Redis（推荐缓存、会话管理）
- 本地文件 → 对象存储（OSS/S3）

## 二、三大功能可行性分析

### 功能 1：推荐系统

#### 需求
根据用户当前任务的内容，推荐相关的学习文章或视频。

#### 技术方案

```
用户创建/查看任务 → 提取任务描述文本 → 向量化(Embedding)
                                           ↓
               文章/视频库 ← 爬虫/用户上传 ← 向量数据库(Milvus/Qdrant)
                                           ↓
                      相似度检索(语义搜索) → 推荐列表
```

#### 关键依赖
- **嵌入模型**：需要将中文任务描述转为向量（如 text2vec-large-chinese 或调用 OpenAI/Claude Embedding API）
- **向量数据库**：Milvus、Qdrant 或 PostgreSQL + pgvector
- **内容源**：用户自行上传 / 内置爬虫 / 调用公开 API

#### 可行性评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 技术难度 | ★★★☆☆ | 嵌入 + 向量检索本身成熟，中文模型表现尚可 |
| 内容来源 | ★★★★★ | **最大难点** — 没有现成的学习内容库，需要用户自建或集成第三方 |
| 运维成本 | ★★★☆☆ | 需要运行向量数据库，增加一个服务 |
| 价值 | ★★★★☆ | 学习场景下内容推荐很有帮助 |

**结论**：技术可行，但内容来源是关键瓶颈。建议 MVP 先做"收藏链接分享"功能，让用户手动积累内容库，再逐步引入自动推荐。

### 功能 2：社交属性

#### 需求
好友系统、查看好友计划、分享文章和视频。

#### 技术方案

```
用户表扩展
  ├── friends 好友关系表 (user_id, friend_id, status)
  ├── shares  分享表 (user_id, content_type, content_id, visibility)
  └── friend_plans 好友计划视图 (权限控制)

权限模型: 私有 / 好友可见 / 公开
```

#### 对当前架构的影响

| 改动点 | 影响 |
|--------|------|
| 数据库 | 需要添加好友关系表、分享表、权限字段，SQLite 可以支持 |
| 后端 API | 新增好友管理、查看好友计划、分享内容等路由 |
| 前端 | 新增好友列表页、好友动态页 |
| 认证 | 当前 JWT 足够，需增加权限中间件 |

#### 可行性评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 技术难度 | ★★☆☆☆ | 标准的社交关系 CRUD，技术成熟 |
| 改造成本 | ★★★☆☆ | 需要改造数据模型和权限系统 |
| 运维成本 | ★☆☆☆☆ | 纯逻辑功能，不增加额外服务 |
| 价值 | ★★★★☆ | 学习社区属性增强用户粘性 |

**结论**：最易实现的功能。在当前架构上可直接扩展，建议优先做。

### 功能 3：AI 代理与陪伴

#### 需求
- AI 回答问题
- 根据用户要求和上传文件，自动生成计划并存入计划表
- 监督和提醒用户完成任务

#### 技术方案

```
用户输入/上传文件
       ↓
LLM API (Claude / GPT)
       ↓
  ┌────┴────┐
  │  结构化  │  →  生成计划/任务 → 写入数据库
  │  对话    │  →  实时回答 → WebSocket 推送
  │  监督    │  →  定时检查任务完成情况 → 推送提醒
  └─────────┘
```

#### 核心改造点

| 改造项 | 原因 | 建议方案 |
|--------|------|---------|
| **引入 LLM API** | 核心能力依赖 | Claude API / OpenAI API |
| **WebSocket** | AI 对话需要实时流式响应 | Socket.IO 或原生 WebSocket |
| **任务调度** | 监督提醒需要定时任务 | node-cron 或 Bull Queue |
| **文件解析** | 上传文件生成计划 | 后端解析 PDF/Word/图片文字 |
| **数据库增强** | 存储对话历史、AI 配置 | SQLite 短期够用，长期用 PostgreSQL |

#### 可行性评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 技术难度 | ★★★☆☆ | LLM API 调用本身简单，难点在结构化输出和可靠性 |
| 改造成本 | ★★★★☆ | 需要 WebSocket、定时任务、文件解析多个新模块 |
| 运行成本 | ★★★★☆ | **LLM API 按 token 计费**，长期使用成本需评估 |
| 价值 | ★★★★★ | AI 生成计划 + 监督提醒是核心增值功能 |

**结论**：技术可行，但改造成本和 API 费用最高。建议作为第三阶段实施。

## 三、分阶段实施路线图

### 第一阶段（1-2 个月）：基础设施升级 + 社交功能

```
1. SQLite → PostgreSQL 迁移
   - 更好的并发性能
   - pgvector 支持后续推荐系统
   - 原生 JSON 查询

2. 文件存储 → OSS/S3
   - 解耦存储和计算
   - 无限扩展空间
   - 后续推荐系统的内容来源

3. 社交功能 MVP
   - 好友系统（搜索、添加、删除）
   - 计划可见性设置（私有/好友/公开）
   - 查看好友计划（只读）
   - 分享链接/笔记到动态

4. 部署
   - Docker 容器化
   - CI/CD 流水线
   - Nginx + PM2
```

### 第二阶段（2-3 个月）：推荐系统

```
1. 内容库建设
   - 用户收藏链接自动抓取标题和摘要
   - 支持用户上传文章/标注视频链接
   - 内置内容分类标签

2. 推荐引擎
   - 集成 Embedding API（或本地模型）
   - 部署向量数据库
   - 任务描述 → 向量检索 → 推荐内容

3. 前端展示
   - 任务面板右侧推荐区域
   - 推荐理由说明
   - 收藏/不感兴趣反馈
```

### 第三阶段（3-4 个月）：AI 代理

```
1. AI 对话
   - WebSocket 实时通信
   - 流式响应显示
   - 对话历史记录
   - 基于学习计划上下文的回答

2. AI 生成计划
   - 解析用户自然语言描述 → 结构化计划
   - 根据上传文件（课程表/大纲）生成计划
   - 自动填充到任务列表
   - 冲突检测与用户确认

3. 监督提醒
   - 定时检查未完成任务
   - 推送通知（邮件/Web 通知）
   - AI 生成学习建议（进度落后时调整计划）
```

## 四、建议优先级

| 优先级 | 功能 | 理由 |
|--------|------|------|
| P0 | 社交功能 | 开发量最小，价值高，为推荐系统铺垫内容基础 |
| P1 | 基础设施升级 | 不升级 SQLite 和文件存储，后续功能无法规模扩展 |
| P2 | AI 代理 | 核心增值功能，但成本较高 |
| P3 | 推荐系统 | 依赖社交功能积累的内容库，可以推迟 |

## 五、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM API 费用不可控 | AI 功能可能太贵无法长期运行 | 设置 token 上限、用户配额、缓存常见问题 |
| 用户内容不足 | 推荐系统无内容可推 | 初始阶段内置一批学习资源链接 |
| 好友功能无人使用 | 社交功能成为废功能 | 先专注 AI 功能，社交作为辅助 |
| SQLite 迁移数据丢失 | 用户数据损坏 | 做好备份导出、迁移脚本充分测试 |
